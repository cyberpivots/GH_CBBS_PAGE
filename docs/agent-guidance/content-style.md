# Content Style

## Voice
- Clear, public-facing, and technically conservative.
- Prefer plain language over hype.
- Explain what CBBS content is about without exposing private implementation details.

## Structure
- Use one `h1` per page.
- Keep headings descriptive and in semantic order.
- Use short paragraphs and meaningful link text.
- Include page summaries in frontmatter for list pages, metadata, and search snippets.

## Claims
- Prefer verified statements such as "This site shares public updates for CBBS."
- Avoid unverifiable statements such as "battle-tested," "secure," "long-range," "certified," or "production-ready" unless source-backed.
- If a claim needs review, keep it in draft content and do not link it from production navigation.
- Product entries must carry a truth-state label: current, proof/simulator, mock scenario, projected, or internal review.
- Agriculture and control copy must stay in the verified lane of records, review, observations, advisory notes, approvals, and guarded command records.

## Media
- Use approved local assets in `public/assets/` or imported assets under `src/`.
- Provide meaningful alt text for informative images.
- Use empty alt text only for decorative images.
- Do not use private screenshots or repository images unless approved.
- Label screenshots and proof captures by provenance. Do not describe simulator/proof images as field deployments.
